VirtualHome — Complete Capability Reference

An interactive simulator of household activities expressed as programs. You write an activity as a short sequence of instructions; a humanoid agent executes it inside a fully-furnished apartment, and the Unity engine renders a photorealistic video — with depth, segmentation, and pose ground truth on tap. Where an action is shown third-person, an aligned egocentric (agent-eye) view of the exact same script is rendered beside it for comparison.

Every clip below was rendered headless on an 8×H200 cluster (Unity 2.3.0, software-GL in Docker). · ← three-simulator overview

The idea

Activities as programs over a scene graph

An activity is a program: a list of [Action] <object> (id) lines (≤2 object args). The environment is a graph — nodes are objects (each with states & properties), edges are relations (ON, INSIDE, CLOSE, FACING, HOLDS). Given (program + graph), VirtualHome runs it in one of two engines:

① Unity Simulator

the downloadable Unity build, driven over HTTP by UnityCommunication. Renders RGB video + depth/segmentation/optical-flow/normals/pose. Everything on this page comes from it.

② Evolving Graph

pure-Python symbolic executor: applies each action's pre/post-conditions and emits the sequence of scene graphs — no rendering, supports an even larger action set.

from unity_simulator.comm_unity import UnityCommunication
comm = UnityCommunication(port='8080')
comm.reset(0)                                  # load apartment 0
comm.add_character('Chars/Female1')            # add the agent
comm.render_script(['<char0> [Walk] <sofa> (1)',
                    '<char0> [Sit]  <sofa> (1)'],
                   recording=True, camera_mode=['AUTO'])   # → video

Representation

The scene graph

Calling comm.environment_graph() returns the whole apartment as JSON — every object with its class, properties, states, bounding box, and its relations to other objects. Programs reference objects by their graph id, and the graph updates as actions execute (a fridge node flips CLOSED→OPEN, a TV OFF→ON).

{ "nodes": [ {"id": 103, "class_name": "fridge",
              "properties": ["CAN_OPEN","HAS_SWITCH","CONTAINERS"],
              "states": ["CLOSED","OFF"], "bounding_box": {...} }, ... ],

Full action vocabulary

Unity-renderable actions: Walk, Run, WalkTowards, WalkForward, TurnLeft, TurnRight, Find, Sit, StandUp, Grab, Open, Close, Put, PutIn, PutBack, SwitchOn, SwitchOff, Drink, TurnTo, LookAt, Touch. Additional symbolic actions in Evolving Graph: Pour, Wipe, Cut, Eat, Read, Lie, Type, Push/Pull, Wash, PlugIn/PlugOut, PutOn/PutOff (clothes), Greet, Sleep.

High-level ↔ low-level

The same task, two control levels

A complex multi-room errand — walk from the living room across to the kitchen and open the fridge — driven two ways. Left: the high-level program [Walk] <fridge> → [Open] <fridge>, where Unity's NavMesh planner routes through the doorways. Right: the same task executed with low-level commands — Unity's returned planner path, converted into a sequence of 1 m [WalkTowards] steps (+ the final [Open]), rendered as one continuous clip. Both reach the identical end state: fridge = OPEN.

high-level program
low-level commands

Walk to the kitchen & open the fridge — high-level vs low-level

left: [Walk]→[Open] (NavMesh planner)  ·  right: 9×[WalkTowards]+[Open] from the recorded planner path

What converts — and what doesn't. A high-level action is a macro whose body is the NavMesh A* planner. It converts to low-level commands by materialising that route: you can read Unity's per-frame planner path (save_pose_data) and replay it as 1 m WalkTowards / TurnLeft / WalkForward steps. Both pipelines deterministically reach the same scene-graph state (fridge OPEN). They are not frame-identical, however: the Unity sim is non-deterministic (the same script run twice gave 88 vs 59 frames, 0 identical), so equivalence holds at the symbolic-state level, not the pixel level.
