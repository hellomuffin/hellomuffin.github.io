The generative model

Task = Scene × Program, filtered by plausibility

A task is a goal program — a sequence of atomic actions [Action] <object> — executed by an agent in a scene. Complexity is composed along four orthogonal axes, and every candidate program is filtered by two gates: precondition validity (enforced by the simulator) and common-sense plausibility (must be imposed on top).

① Scene complexity

7 base apartments · 50 hand-authored variants · unlimited via procedural_generation(). 4–5 rooms and 320–444 objects each.

② Navigation route

# rooms traversed (1 → 3+), number of legs/sub-goals, path length, doorways threaded. See the floor plan & trajectory below.

③ Manipulation depth

Chain length L, number of distinct objects, container nesting (open→take→close), state changes, and ordering constraints.

④ Plausibility tier

type-valid ⊃ semantically-sensible ⊃ goal-coherent. The plausible space is a small, structured fraction of the syntactic space.

The real constraint

Common-sense plausibility: three nested gates

① Type-valid

Simulator-enforced. The action's required property must hold and pre-conditions must be met (can't Open a non-openable; can't Put without first holding; must be CLOSE). Prunes all type-invalid combinations automatically.

② Semantically sensible

Must be added. Object↔receptacle compatibility: Put(food, fridge) ✓ vs Put(food, toilet) ✗; Put(plate, dishwasher) ✓ vs Put(plate, bed) ✗. The sim allows both — common sense forbids the second.

③ Goal-coherent

The narrowest gate. The chain must achieve a recognizable everyday goal — tidy up, prepare a snack, get ready for bed — not a random valid sequence. This is what makes a task a task.

The plausible space is a small, structured subset of the ~hundreds-of-thousands of type-valid programs. Generating good tasks = sampling the syntactic space, then filtering by (②) a semantic object–receptacle compatibility model and (③) a goal template / LLM plausibility check. This is exactly the recipe behind large LLM-generated task sets (e.g. PARTNR's 100k).

Beyond open→take→close→place

Chain families — the full space of multi-step structure

A chain is any sequence where each action's effect satisfies the next action's precondition. Every action has preconditions (required object properties/states) and effects (state changes), so valid chains are paths through a state-transition machine over object & agent states. open→grab→close→put is just one path (the "retrieve & relocate" family). The action + property set affords many more:

The generative view. Draw a graph whose nodes are (object-state, agent-state) and whose edges are actions (an edge exists when the action's precondition holds and it produces the new state). Every directed walk through this graph is a valid chain. The families above are the goal-shaped walks; open→take→close→place is one of them. New families appear automatically as you include more of the property set — POURABLE, CUTTABLE, CLOTHES, HAS_PLUG, READABLE, DIRTY/CLEAN — most of which live in the symbolic Evolving-Graph engine, with the Unity-renderable subset noted above.
How diverse

Diversity dimensions

object identity (314 classes) room & layout (8 room types) receptacle type (surface / container) action mix (open/switch/sit/pour…) navigation depth (1–3+ rooms) temporal ordering constraints spatial constraints (on/in/next-to) single vs multi-agent
Bottom line. The raw compositional space is effectively unbounded (≳10⁵ tasks at depth ≤3 per apartment, exponential beyond, × unlimited scenes). The useful space is the plausibility-filtered subset — large but structured — which is exactly what a benchmark should sample along these four axes.
