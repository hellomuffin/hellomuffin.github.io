Three Simulators for Proactive Streaming Agents

A hands-on guide to Habitat 3.0, Overcooked-AI, and Craftium — the candidate engines for an Interactive Online Policy Evaluation benchmark, where a user policy acts in a world and a proactive assistant only watches the user's RGB stream and decides whether, when, and what to communicate.

Every video below was generated headless on a GPU cluster by the task code in this repo.

Why these three engines

The benchmark needs a user policy that physically acts, a separate assistant policy that observes only the egocentric/2D RGB stream, and outcome metrics (did the intervention help?). The three engines span the fidelity/scale spectrum that question requires:

HABITAT 3.0

Photorealistic 3D homes. Humanoid + robot agents, egocentric RGB/depth, real manipulation. Highest visual & physical fidelity; closest to real wearable-camera deployment.

OVERCOOKED-AI

Real-time 2D coordination. Two cooks, sequential recipe, bundled human-play data. Cleanest testbed for when to speak, intent prediction, and interruption cost.

CRAFTIUM

Open-world voxel sandbox. Huge compositional action space, first-person RGB, fully moddable tasks. Best for long-horizon, open-ended assistance.

HABITAT 3.0

Habitat 3.0 — photorealistic embodied homes

A high-performance 3D simulator (FAIR) with photorealistic indoor scenes, rigid-body physics, and articulated agents. Habitat 3.0 adds native human (SMPL-X) avatars and multi-agent co-habitation — so one body can be the user while another agent assists. We drive long-horizon household tasks and render the robot's egocentric head camera at every step.

Egocentric navigation (realistic RGB)

A first-person tour of a furnished room — the exact visual stream the assistant consumes.

SetTable  ✓ solved

11-step PDDL plan: navigate → open cabinet → pick bowl → place → open fridge → pick fruit → place → stop. Objects land within ~1 mm of their goals.

TidyHouse  ✓ solved

20-step plan relocating 5 misplaced objects to their correct receptacles.

PrepareGroceries  ✓ solved

12-step plan moving 3 grocery items between fridge, counter and cabinets.

wipeBlackboard  ✓ custom-authored

Habitat has no native blackboard, so we authored one: a board with 24 dirt marks; a scripted sponge sweep wipes them away. 24/24 cleaned.

The board starts covered in dirt marks (left) and ends clean (right) — an outcome the benchmark can measure directly.

What it can simulate

Multi-step tasksHome Assistant Benchmark (SetTable, TidyHouse, PrepareGroceries) defined in PDDL with explicit stage-goals; Social Rearrangement & Social Navigation (robot + humanoid); PARTNR (100k natural-language collaboration tasks).
Action spaceNavigation (discrete + continuous base velocity); 7-DOF arm + suction/magic grasp; high-level oracle skills (OracleNav, PddlApply); humanoid SMPL-X joint & pick actions.
AgentsSMPL-X humanoids (~12 bodies); robots Spot / Fetch / Stretch; native multi-agent.
ObservationsEgocentric RGB / depth / semantic / panoptic; GPS+compass; object detectors.
DiversityHSSD (200+ scenes), ReplicaCAD, HM3D, MP3D; YCB + articulated objects (fridges, drawers).
How we solve them: an oracle PDDL executor runs each task's grounded solution plan via PddlApplyAction and signals completion with rearrange_stop — yielding pddl_success=True on all three HAB tasks. This is exactly the kind of competent "user policy" the benchmark needs target trajectories from.
OVERCOOKED-AI

Overcooked-AI — real-time 2D coordination

The canonical human-AI coordination testbed (Carroll et al., NeurIPS 2019). Two cooks share a kitchen and must jointly run a sequential recipe under time pressure: onion → pot ×3 → cook → wait → plate → deliver, looped for reward. We run goal-directed GreedyHumanModel cooks across diverse layouts and render the 2D RGB stream.

cramped_room

Shared pot; tight space. 2 soups delivered.

asymmetric_advantages

Players have different access to stations. 3 soups.

counter_circuit

Ring layout encouraging hand-offs over a counter.

coordination_ring

Needs true cooperation — greedy agents alone score 0, which is exactly where an assistant would help.

forced_coordination

Neither cook can finish alone — a pure coordination stress-test.

A delivery moment

Chef plating a finished soup (pot timer + onions visible).

What it can simulate

Multi-step taskThe ordered soup-delivery loop, repeated over the episode horizon; branching recipes (onion vs tomato) and bonus orders.
Action space6 actions/player: N/S/E/W, STAY, and a context-dependent INTERACT (pick up / drop / fill pot / start cook / plate / deliver). Stepped as a joint action.
AgentsScriptable user policies: GreedyHumanModel, Random, FixedPlan, AgentFromPolicy; bundled human-human play datasets.
ObservationsRendered RGB frame, plus featurized vector and lossless grid-tensor encodings.
Diversity~50 layouts, each encoding a different coordination challenge, + programmatic generation.
The layouts that score 0 with two independent greedy cooks (forced_coordination, coordination_ring) are the most interesting for us: they isolate the value of a proactive assistant that resolves coordination.
CRAFTIUM

Craftium — open-world voxel sandbox

A Minecraft-like engine (built on Luanti, ICML 2025) exposed through a Gymnasium API with a first-person RGB stream — no Java, fully Lua-moddable. Best fit for a large compositional action space and long-horizon, open-ended tasks.

ChopTree — first-person forest ✓ rendered

The user walks through a procedurally generated forest; the agent is rewarded for chopping wood. Real headless render on the GPU cluster.

PlantTree — custom task ✓ rendered

A new task authored as a Lua mod on the forest world: the user is given saplings and rewarded for planting them. Sibling tasks BuildHouse (reward per wood block placed) and ProcDungeons (procedural multi-room dungeon) use the same pipeline and capture during good GPU render windows.

What it can simulate

TasksBuilt-in: ChopTree, ProcDungeons, Speleo, SpidersAttack, Room, OpenWorld. Custom (authored here): PlantTree, BuildHouse. Plus the open-ended craft/build/survive dependency chain. Fully Lua-extensible; PettingZoo multi-agent.
Action spaceRaw: ~21 binary keys (move/jump/sneak/dig/place/drop/inventory/hotbar 1-9) + 2-D continuous mouse look. Per-task Discrete wrappers for convenience.
ObservationsFirst-person RGB (configurable res), optional grayscale and 3-D voxel; player pos/vel/pitch/yaw in info.
DiversityProcedural worlds & dungeons, day/night, mobs, biomes; the entire Luanti modding ecosystem.
Rendering note (engineering reality). This headless cluster exposes only NVIDIA EGL (no software-GL driver, no X server, no root). Luanti's offscreen GL init is racy here (0–25% per launch, varying with shared-GPU state), so we rebuilt the engine from source, added a retry-on-blank wrapper, and a rootless-Xvfb fallback. ChopTree renders cleanly; the other tasks capture opportunistically during good GPU windows.

Run it yourself

Each engine lives in its own micromamba environment under sims/. Representative commands:

# Habitat 3.0 — solve a household task with the oracle, render egocentric RGB
micromamba run -n habitat python sims/habitat/task_hab_oracle.py set_table
micromamba run -n habitat python sims/habitat/task_wipe_blackboard.py

# Overcooked-AI — full soup-delivery loop + multi-layout showcase
SDL_VIDEODRIVER=dummy micromamba run -n overcooked python sims/overcooked/task_soup_delivery.py
SDL_VIDEODRIVER=dummy micromamba run -n overcooked python sims/overcooked/showcase.py

# Craftium — first-person tasks (uses a retry-on-blank render wrapper)
micromamba run -n craftium-build python sims/minecraft/capture_choptree.py
micromamba run -n craftium-build python sims/minecraft/capture_all.py   # PlantTree / BuildHouse / ProcDungeons

Full research notes and reproduction steps are in each engine's RESEARCH.md and SETUP.md, and a status overview in sims/TASKS.md.
